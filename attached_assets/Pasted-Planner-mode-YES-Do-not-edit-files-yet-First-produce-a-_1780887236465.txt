3. Dynamic price filter chips
   Replicate the UIB `buildDynamicChips` behaviour.

Relevant UIB functions:

* `buildDynamicChips`
* `roundBoundary`
* `fmtRange`
* `applyPriceRange`
* `PriceRangeSelector`

Anchor price logic:

* `qty = Math.max(1, Math.round(item.quantity ?? 1))`
* if `qty > 1` and `item.unitEstimatedPrice > 0`, use `unitEstimatedPrice`
* otherwise use `item.estimatedPrice`
* mobile improvement allowed: if `qty > 1`, `unitEstimatedPrice` is missing, and `estimatedPrice` exists, derive unit anchor as `estimatedPrice / qty`
* If no usable anchor exists, show only:

  * `Any`
  * `Custom`

Rounding logic must match UIB:

* anchor `< 50` → nearest `5`
* anchor `< 150` → nearest `10`
* anchor `< 500` → nearest `25`
* anchor `< 1500` → nearest `50`
* otherwise → nearest `100`

Chip IDs:

* `any`
* `lower`
* `around`
* `higher`
* `premium`
* `custom`

Price bands:

* `Any` — no min/max
* `Lower` — `0.25x` to `0.75x` anchor
* `Around` — `0.75x` to `1.25x` anchor
* `Higher` — `1.25x` to `2.0x` anchor
* `Premium` — `2.0x+`
* `Custom` — user-entered min/max

Mobile labels should be consumer-friendly:

* Any
* Lower $60–$100
* Around $100–$170
* Higher $170–$270
* Premium $270+
* Custom

Use NZD formatting by default but keep the logic country-ready for future rollout.

6. Result processing
   Relevant UIB functions:

* `parsePrice`
* `assignMatchType`
* `applyPriceRange`
* `isPartOrSupportResult`
* `isWeakResultSet`
* `buildFallbackQuery`
* `fetchResults`
* `fetchPanelResults`

Two-pass fallback search:

* First pass searches the exact built/user query.
* After pass 1, call `isWeakResultSet`.
* A result set is weak if:

  * zero results, or
  * fewer than 2 priced full-product results, or
  * more than 50% of results look like support/parts/manual pages.
* If weak, call `buildFallbackQuery`.
* `buildFallbackQuery` strips model-code tokens using equivalent logic to:

  * `\b[A-Z][A-Z0-9]{4,14}\b`
* Run pass 2 only if the fallback query differs and is meaningful.
* Use pass 2 only if it produces more priced full-product results.
* Show a small user-facing notice:

  * `No strong exact-model listings found — showing comparable replacements.`

Part/support/manual detection:
Replicate the UIB `PARTS_SUPPORT_PATTERNS` concept. De-emphasise results that look like:

* manual
* user guide
* driver
* firmware
* download
* support
* spare part
* replacement part
* repair kit
* battery
* charger
* power adapter
* cable
* adapter
* screen replacement
* hinge
* motherboard
* keyboard replacement
* dock
* warranty
* support URLs such as `/support/`, `/drivers/`, `/manuals/`, `/downloads/`

Do not necessarily hide these results, but visually de-emphasise them:

* dimmed card
* small badge: `May be part/support`
* avoid making them feel like the recommended default
* disable or de-prioritise `Use as replacement price` if no valid product price exists
